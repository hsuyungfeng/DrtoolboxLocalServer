# Phase 2 Plan 02: LINE Bot Integration (Wave 2)

## 概述
Wave 2 implements LINE patient messaging by building a webhook endpoint, message router that classifies intent, RAG-based response sender, and comprehensive tests. Builds on Wave 1 HIS integration to create cohesive patient communication system.

---

## ✅ Success Criteria (Wave 2 specific)

From Phase 2 ROADMAP — these gates apply to Wave 2:

- [ ] **SC-2**: LINE bot responds to 100% of incoming messages within 5 seconds
- **Why this matters**: Patients expect sub-5s response; >5s feels broken; tests must validate latency SLA
- **Verification method**: Performance test in Task 8 measures end-to-end latency from message receipt to LINE send

---

## 📋 Task Breakdown (Wave 2)

### Task 5: LINE Webhook Endpoint & Message Parsing

**Purpose**: Accept incoming LINE messages and parse user intent

**Deliverable**: `src/api/routes/line_bot.py` + Flask Blueprint

**Acceptance Criteria**:
- POST /api/line/webhook receives LINE Events API messages
- Parses message type (text, image, etc.) and user ID
- Validates LINE signature (uses LINE channel secret)
- Implements retry handler for failed deliveries (LINE expects 200 OK within 3s)
- Logs all incoming messages for audit trail
- Registers Blueprint in src/api/app.py

**Estimate**: M (Medium, ~2-3 hours)

**Blockers**: LINE channel ID, channel secret (clinic provides — use mocks for testing)

**Related Decisions**: D-03

**Related Requirements**: COMM-01

**Implementation notes**:
- Use LINE SDK: `from linebot import LineBotApi, WebhookHandler`
- Signature validation: HMAC-SHA256 using channel secret
- Message type detection: text, image, location, template
- 3s timeout requirement: respond quickly; defer heavy processing to async queue if needed

---

### Task 6: Message Routing Logic (RAG vs. Escalation)

**Purpose**: Classify incoming patient messages and route to appropriate handler

**Deliverable**: `src/services/message_router.py`

**Acceptance Criteria**:
- Uses hybrid query intent classification (reuse logic from scripts/hybrid_query.py)
- Default route: send to RAG (D-03)
- RAG returns response + confidence score
- If confidence < 60% (D-04), flag for escalation instead of sending to patient
- Logs routing decision and confidence score for all queries
- Edge case: malformed/abusive messages → automatic escalation

**Estimate**: M (Medium, ~2-3 hours)

**Depends on**: Task 5 (needs message parsing infrastructure)

**Related Decisions**: D-03, D-04

**Related Requirements**: COMM-02

**Implementation notes**:
- Reuse `scripts/hybrid_query.py` intent classification or call existing RAG API
- RAG API endpoint (from Phase 1): GET /api/rag/query → returns {response, confidence}
- Confidence threshold: < 60% triggers escalation flag
- Log format: {user_id, message_text, intent, confidence, routing_decision}

---

### Task 7: LINE Response Sender & Error Handling

**Purpose**: Send RAG responses back to patient via LINE, with fallbacks for errors

**Deliverable**: `src/services/line_responder.py`

**Acceptance Criteria**:
- Sends response message to patient via LINE Messaging API
- Includes source citations (from RAG) when available
- If RAG unavailable, sends "We're having trouble. Staff will follow up soon."
- Retries failed sends (LINE API transient failures)
- Logs all outbound messages
- All responses sent within 5 seconds of incoming message (SC-2)

**Estimate**: S (Small, ~1-2 hours)

**Depends on**: Task 6 (receives routed messages)

**Related Requirements**: COMM-02

**Implementation notes**:
- Use LINE SDK: `line_bot_api.push_message(user_id, message)`
- Citation format: append source document/page at end of response
- Fallback message: pre-written error message for RAG unavailability
- Retry strategy: exponential backoff for transient LINE API failures (max 3 retries)
- Latency target: response sent within 5s of receipt (Task 8 validates)

---

### Task 8: LINE Integration Tests & Response Time Validation

**Purpose**: Verify LINE bot end-to-end with response time SLA

**Deliverable**: `tests/test_line_integration.py`

**Acceptance Criteria**:
- Mock LINE API (simulate incoming messages)
- Test normal flow: patient message → RAG query → response sent
- Test escalation flow: low-confidence message → escalation instead of send
- Performance test: verify 100% of responses sent in <5 seconds (SC-2)
- Test error handling: RAG unavailable, LINE API down
- Success Criterion SC-2 verified

**Estimate**: M (Medium, ~2-3 hours)

**Depends on**: Task 5, Task 6, Task 7

**Related Requirements**: COMM-01, COMM-02

**Implementation notes**:
- Mock library: `unittest.mock` or `pytest-mock`
- Test scenarios:
  1. Normal: message → RAG (confidence 80%) → response sent (latency <5s)
  2. Escalation: message → RAG (confidence 45%) → escalation flag, no response to patient (latency <5s)
  3. RAG error: message → RAG error → fallback message sent (latency <5s)
  4. LINE API error: retry logic works, message eventually sent or logged
- Latency measurement: use `time.time()` at message receipt and response send; assert all < 5s
- Coverage: aim for 80%+ line coverage on Tasks 5-7

## 🔧 Key Decisions (Locked from 02-CONTEXT.md)

### D-03: LINE Default Route to RAG
- **Why**: Patients ask general medical questions more often than clinic-specific queries
- **How**: Message router defaults all messages to RAG; only escalate if confidence < 60%
- **Implementation**: Task 6 routes 100% to RAG unless confidence threshold triggers escalation

### D-04: Escalation Threshold < 60%
- **Why**: Safety-critical; avoid low-confidence medical advice to patients
- **How**: RAG confidence score < 60% → flag for staff instead of sending response
- **Implementation**: Task 6 checks confidence; Task 7 only sends if >= 60%

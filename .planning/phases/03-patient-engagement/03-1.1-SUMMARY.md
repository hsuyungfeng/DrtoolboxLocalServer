# Task 1.1: Extend clinic.db Schema for Patient Intake — Summary

## Objective

Extend the existing clinic.db SQLite schema with three new tables (`patients`, `appointments`, `line_user_mapping`) and supporting indices to enable patient intake collection, appointment scheduling, and LINE user linking.

## Deliverables Completed

### Tables Created

1. **patients** — Core patient demographic and medical history
   - Columns: patient_id (PK AUTO), name, phone, email, dob, medical_history, allergies, created_at, updated_at, created_by, updated_by
   - Constraints: All required fields NOT NULL except medical_history/allergies/updated_at/updated_by
   - Audit columns: created_at (DEFAULT CURRENT_TIMESTAMP), updated_at, created_by, updated_by

2. **appointments** — Appointment scheduling and tracking
   - Columns: appointment_id (PK AUTO), patient_id (FK), appointment_date, status (DEFAULT 'pending'), created_at, updated_at, created_by, updated_by
   - Constraints: Foreign key to patients(patient_id), all required fields NOT NULL except updated_at/updated_by/created_by
   - Audit columns: created_at, updated_at, created_by, updated_by

3. **line_user_mapping** — Patient-to-LINE user linkage
   - Columns: mapping_id (PK AUTO), line_user_id (UNIQUE NOT NULL), patient_id (FK), created_at, linked_by
   - Constraints: Foreign key to patients(patient_id), UNIQUE on line_user_id for one-to-one mapping
   - Audit columns: created_at (DEFAULT CURRENT_TIMESTAMP), linked_by

### Indices Created

1. **idx_patients_phone_email** — UNIQUE composite index on (phone, email)
   - Purpose: Enforce phone+email uniqueness for idempotency (D-03 requirement)
   - Deduplication: Prevents duplicate patient records from multiple form submissions

2. **idx_patients_created** — INDEX on created_at
   - Purpose: Fast queries for time-range filters (e.g., "patients created today")
   - Performance: O(log n) lookup instead of O(n) table scan

3. **idx_appointments_patient** — INDEX on patient_id
   - Purpose: Fast lookup of all appointments for a given patient
   - Performance: Critical for dashboard queries listing patient's upcoming appointments

4. **idx_line_user_mapping_patient** — INDEX on patient_id
   - Purpose: Fast lookup of LINE mappings for a patient
   - Performance: Supports reverse lookup (given patient_id, find associated line_user_id)

### Global Configuration

- **PRAGMA foreign_keys = ON** — Enabled at top of schema file
- Already present in original schema, verified in extension

## Implementation Notes

### Design Decisions

1. **Audit Column Pattern**: All tables follow consistent pattern (created_at, updated_at, created_by, updated_by) matching existing clinic tables (clinic_staff, clinic_equipment, etc.)

2. **UNIQUE Composite Index**: Phone+email composite enforces idempotency per D-03. Prevents duplicate patient records when same person submits form twice (with identical contact info).

3. **Foreign Key Relationships**:
   - appointments.patient_id → patients.patient_id (one-to-many: one patient has many appointments)
   - line_user_mapping.patient_id → patients.patient_id (one-to-one: one LINE user maps to one patient)

4. **Status Field**: appointments.status defaults to 'pending' per Phase 3 spec (workflow: pending → approved → completed)

5. **LINE User Mapping UNIQUE Constraint**: line_user_id is UNIQUE (not composite) to prevent one LINE user being linked to multiple patients (privacy/routing clarity)
